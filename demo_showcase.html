1. 把 80 存到 budget
2. 仅当 budget 当前值大于 100 时，才把它清零

更关键的问题：在接入生产之前，你怎么系统性地知道它还会以哪些方式失败？

Why · 为什么做这件事

能力有人测，失败没人管

Capability is benchmarked everywhere; failure is nobody's job

1Agent 正在接管真实操作

工具调用型 Agent 开始被接入订单、运维、资金流。失败不再是"答错一道题"，而是不可逆的真实损失。上一页那种"违背保护条件"的失败，在生产里就是事故。

2评测体系只看能力上限

几乎所有 benchmark 回答的都是"它能做什么"。而失败模式是长尾分布：什么条件下会覆写受保护状态？什么指令会让它死循环？没有系统性的编目，只有出事后的个案归因。

3失败案例本身是资产

一条可复现的失败 = 一条回归测试（升级模型后跑一遍）+ 一条选型依据（哪个模型在哪类任务上不可信）+ 一条训练数据（针对性修复）。它值得被生产、验证、定价。

问题在于：挖失败是对抗性搜索，长尾意味着单一团队永远覆盖不完。
要让全网的人持续地、有报酬地帮所有人找失败——这需要一个开放的激励市场。

这正是 Bittensor 子网这个形态存在的意义。↓

Why Bittensor · 为什么是子网

这条子网在做什么？三件事

Reproducible failures in, code-verified replay, rewards by long-term quality

矿工提供稳定可复现的翻车场景，验证者用代码自动复查这些翻车，奖励按矿工长期贡献的翻车质量来分配。

Miners矿工在做的事

不是在"算题"或"回答问题"，而是给系统找能稳定复现的翻车场景。一个合格的场景要同时有三样：

  • 一套明确的任务环境——别人能原样加载起来的那种
  • 一段判分代码——自动判断这次任务是成功还是失败
  • 一条参考解法——证明这题不是无解，只是目标 agent 会在这里犯错

Validators验证者在做的事

确认这个翻车是真实的、可重复的，不是矿工瞎编的。按同一套流程检查：

  • 先用参考解法跑一遍，确认这题确实能做对
  • 再让目标 agent 同样条件跑一遍，看它是不是稳定会在这里翻车
  • 最后由判分代码自动给出"确实失败"的结论

全程没有人为打分，也没有"让另一个大模型来当裁判"。

Incentive奖励怎么发

三步走：

  • 每个通过验证的场景得一个分数——难度高、足够新颖的，分数更高
  • 不看单次爆分，看你一段时间的平均表现——奖励持续挖好案例的人，不奖励偶然
  • 平均分换算成占比：占比越大，从子网总奖励里分到的越多

具体公式和数字，下一页逐步算给你看。

这条子网能不能站住脚，取决于一件事：打分够不够硬

只要打分是主观的、不可复现的，或者容易被人"玩规则"，激励就会导向垃圾产出——这是许多子网失败的共同原因。FaultLine 把它定为首要约束：所有判决都落在明确的代码断言上，任何人都能重放环境、复现整条判断链路；代码判不准的指标（语气、有用性、风格）宁可暂时不评——少评一点，换每一项结果都可验证。

定价公式

score = 0.4 × difficulty + 0.6 × novelty
difficulty 0.4novelty 0.6

难度：绊倒了注册表里多少个目标模型（按权重计）。新颖度：这类失败此前被发现过几次，按 1/√n 衰减——首发满分，重复递减。

为什么新颖度占大头（0.6）？题库的价值在覆盖面：某类失败的第 1 条值满分，第 100 条几乎不值钱；而且新颖度权重越高，换皮作弊被罚得越疼（数字见"参数校准"页）。为什么用 1/√n 缓衰减、不直接归零？同类失败的第 2、3 条仍有交叉验证价值——缓衰减保留深挖一类的动力，又压死无限刷量。

两道门挡住什么

  • 门 1 挡"无解题"：出一道谁都做不出的题让 AI 必挂？参考解过不了自己的 checker，直接 0 分
  • 门 2 挡"空口宣称"：说它失败不算数，validator 在钉死的 (模型 × harness × 工具箱) 上当场复现才算数
  • 恶意代码：checker 在沙箱裸环境运行，import os 之类当场报错 0 分
为什么门 1 必须矿工自证？从外面看，"超难的题"和"根本无解的题"长得一模一样，人工逐题审查无法规模化——唯一可扩展的办法是出题人自带机器可验证的证明。为什么门 2 要钉死版本？模型、harness、工具箱任何一个变了，都等于换了一台测量仪器，失败在不同仪器间不可比——我们实测过：一次一行的 prompt 修复，让整个 tier 的失败分布整体移动。

Anti-manipulation · 评分能被操纵吗

计费的每个输入，都不由矿工的文字决定

Every billing input is validator-derived — miner strings never count

攻击 1：同一发现，换个写法再提交

把 checker 改名、重排、换断言标签，伪装成新发现。识别依据是行为而非文本：validator 给每个 checker 做两组"体检提问"，把回答模式规范化后取哈希——

  • 共享探针：一组固定的标准状态，所有 checker 答同一份卷子，跨提交可比
  • 案例派生探针：围绕这道题自己的标准答案状态做扰动——删一个键、改一个值、换个错误答案，看 checker 怎么反应
原版指纹 2a7ab814… 改写版指纹 2a7ab814…

指纹相同 → 新颖度在类别衰减之上再乘 0.25：首发 1.0 → 改写重交 0.1768

为什么要第二层探针？只有共享探针时，凡是探针问不到的键，所有 checker 都表现为"处处失败"——行为不同的题会被误判成重复，冤枉诚实矿工（这个盲区是我们自己的体检工具先抓出来的，见"参数校准"页）。派生探针让每道题在自己的答案附近被审视；而改写者抄的是同一道题，派生出的探针也一模一样，照样撞车。想躲过它？唯一办法是改动任务环境本身——那已经不是换皮，是出新题，而新题的新颖度照样按类别衰减。

攻击 2：自创失败类别，开新计费桶

return (..., 'ZZ_AMAZING_NEW_FAILURE')  # 想开新桶

失败类别表是封闭且带版本号的（18 类：状态纪律、字符感知、工具误用、规划、上下文……）。未知前缀一律计入 UNCLASSIFIED，新颖度 × 0.5，得 0.7 而非 1.0。

作弊路径旧设计现在
改写文本刷新颖可行指纹识破 ×0.25
自由文本开新桶无限封闭表 + ×0.5
改环境躲指纹—类别衰减兜底
无解题 / 恶意代码00（结构性）
为什么类别表必须封闭？计费单位如果能被自由文本创造，新颖度就能无限铸造——旧设计里矿工随手写个标签就能开新桶。封闭表把铸造上限钉死在类别数；真发现新类别，走治理流程扩表：附证据、过 7 天公示，全网受益。正路敞开，捷径封死。

已知残留风险也写在 README 里：共享探针是公开的，理论上可被针对性特判——派生探针已把规避成本抬到"改环境"级别，validator 私有探针轮换仍在路线图上。弱点自己先讲。

Registry · 目标模型过时了怎么办

钉死才能复现，滚动才不过时

Pinned instruments in a rolling window, ranked by measured pass rate

复现要求目标钉死（模型 × harness × 工具箱），但钉死的目标会过时；更新太快，矿工的存量案例又一夜作废。解法：同时对一组带权重的目标评分，权重随版本滚动。

排序依据是实测通过率，不是参数量——我们实测过 360M 模型通过率是 500M 模型的 3 倍，按大小排 tier 会算出负难度。生命周期：active → deprecated ×0.5 → retired 0。

难度 = 挂掉目标的权重和

  • qwen2.5-1.5b（最强）通过 w=0.500
  • smollm2-360m 失败 w=0.333
  • qwen2.5-0.5b 失败 w=0.167

difficulty = 0.333 + 0.167 = 0.5

score = 0.4 × 0.5 + 0.6 × 1.0 = 0.8

只绊倒旧模型拿部分分；绊倒最新最强模型才拿满——矿工的注意力被自动引向前沿。

纪律性约束：注册表更新从不追溯重算已发放的 emission——历史收入安全，激励才稳定。

Governance · 规则谁说了算

Owner 可以提议，不能突袭

Timelocked parameters; machine-checkable phase transitions

评审的原话："Owner 调整评分参数偏向某些矿工怎么办？"——这是所有子网共有的中心化 oracle 问题。FaultLine 的回答不是承诺，是让违规修改在代码层面不生效。

Calibration · 0.4 / 0.6 凭什么

参数不是拍脑袋定的，也不能拍脑袋改

Beta defaults · offline weight sweep · pre-registered tripwires

Before launch上线前：离线重放

拿已归档的 45 案例 × 3 模型矩阵，在 4 组候选权重 × 20 个随机提交顺序下重放全部计分，看矿工策略的收益排序怎么变。

结果：0.4/0.6 处在排序稳定区；最担心的"低难度题靠新颖度混进高分区"，在最偏新颖度的 0.3/0.7 下也是 0 例。

为什么要 20 个顺序？新颖度按提交顺序衰减，同一批案例换个顺序、分数就不同——单一顺序得出的结论是运气，不是证据。

The hard reason为什么难度权重不敢调高

换皮案例的新颖度会被指纹罚砸掉，但难度分毫发无损。难度权重越高，作弊者保住的分越多：

权重组合换皮案例还能拿
0.3 / 0.7原分的 42%
0.4 / 0.6（现行）原分的 51%
0.6 / 0.4原分的 67%
这就是 0.4 不往上调的硬理由：难度权重每加一分，第 4 页那个反作弊惩罚就钝一分。参数之间不是独立旋钮，是相互咬合的齿轮。

After launch上线后：预注册的改动条件

什么情况下才讨论改权重，提前写进参数文件：

  • 低难度案例混进高分区超过 10% → 讨论 0.5/0.5
  • 单一类别吃掉 50% 收益、且新指纹停滞 → 提高新颖度权重或加陡衰减
  • 注册表 / 类别表任何变更 → 必须重跑 sweep 才能上线
为什么要预先写死？防止"先想改参数，再找数据凑理由"。触发了也不能立刻改——照样走 7 天公示 + 附证据。

这套体检的第一个战果：抓到的是我们自己

第一次 sweep 报出 23/45 重复指纹，且这个数字对四组权重完全不敏感——对权重不敏感，说明不是激励问题，是测量问题。追查发现：v1 探针集从不触碰生成 checker 断言的那些键，行为不同的 checker 全部坍缩成同一个"处处失败"指纹，上线会把诚实矿工的不同案例误判成重复。修复（案例派生探针，上上页）后重跑：23 → 0，同案例的改写依旧被识破。第一个被抓住的缺陷出在我们自己的测量工具里——这套机制正常工作的样子，就该是这样。

诚实的限定：离线重放用的是激励存在之前挖出的案例，真实矿工会适应规则——所以 sweep 只能划界，不能预测均衡。这正是 tripwire + 公示期存在的原因。

One more thing · 回到开场

还记得开场那次翻车吗？

那个把自己被要求保护的值清零的 agent——它的翻车现场，现在是题库里编号 seed_002 的场景。下一个新模型发布时，跑一遍题库，就知道这个错它还犯不犯。这就是这条子网在积累的东西。

今天翻车场景越攒越厚

每一条通过验证的翻车，都永久成为一条公共回归测试——全网的模型升级，都多一道免费的安检。

持续擂台上永远是新模型

目标滚动更新、按实测定权重，题库不会烂在过时模型上——矿工的注意力始终对准前沿。

然后用失败教 AI 预判失败

标本攒够（8 类 × 20 指纹 × 2 tier，进度公开可查），开启第二阶段：训练能提前预判翻车的模型——从编目失败，到预防失败。

AI 的能力每个月都在刷新，而它的失败至今没有一份系统性的账本。
FaultLine 想做的，就是把这份账本建立起来——而且让它不依赖任何人的诚实。
